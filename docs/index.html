Submission note

Brief smoothing implementation note

The source code is available at github.com/devanshujain95/smooth-scribble. The drawing implementation lives in the WebView's HTML5 Canvas runtime, and the React Native layer only sends low-frequency control commands.

01

Drawing and smoothing logic

The drawing surface is an HTML5 Canvas hosted through the react-native-webview library. Pointer events are handled inside the canvas runtime, which keeps high-frequency touch data away from React Native state and avoids sending every point over the bridge.

  • Point thinning ignores noisy duplicate points closer than 1.5px, while still accepting a point after 24ms so slow strokes do not feel stuck.
  • A velocity-aware low-pass filter smooths slow movement more and fast movement less, so careful lines stabilize without making fast scribbles visibly trail behind the finger.
  • Quadratic midpoint curves render the accepted smoothed points as rounded Bezier segments instead of sharp polylines.
  • Fast pointer moves are batched with requestAnimationFrame, and coalesced pointer events are consumed when the browser provides them.
  • Completed strokes are stored in WebView-side history for undo/replay, not in React Native state, while active strokes are drawn incrementally instead of redrawing the full canvas on every move.
02

Tradeoff and real-device performance consideration

The main tradeoff is choosing a WebView canvas for a compact, inspectable exercise implementation instead of a native renderer.

This keeps the smoothing code easy to review and avoids drawing libraries, but on real devices I would pay close attention to WebView rendering cost, memory growth from long sessions, and touch latency on older iPhones. If profiling showed stroke volume or export quality becoming the bottleneck, I would move the renderer closer to native while keeping the same custom smoothing model.

An example of a touch-heavy or interactive feature I built

I built a very similar canvas-heavy feature for one of my apps, LovelyCouple, where the user could draw on a shared canvas and see that drawing reflected in an iOS widget in near-real time.

  • In LovelyCouple, the canvas state was shared with another connected user through WebSockets, so both users could stay in sync while interacting with the drawing experience.
  • I also added an iOS widget flow where the latest drawing could be surfaced on the widget tile, giving the user a glanceable version of what was created in the app.
  • For that production-style feature, I used React Native Skia because it made real-time canvas rendering, state management, and widget snapshot generation simpler to manage.
  • For this exercise, I intentionally used a custom WebView HTML5 Canvas implementation because you specifically asked to avoid drawing libraries and show my own smoothing logic.
03

How I would improve a laggy drawing experience on older iPhones

  • First I would profile real devices and separate input latency, rendering cost, bridge traffic, memory pressure, and JS-thread work.
  • I would keep pointer processing close to the drawing surface instead of sending every touch point through React Native state.
  • I would thin duplicate points, batch drawing with code requestAnimationFrame, use coalesced pointer events when available, and cap stroke history.
  • I would tune smoothing constants against real touch latency so fast strokes do not trail behind the finger.
  • If stroke volume, export quality, or long sessions became the bottleneck, I would evaluate a native renderer while keeping the smoothing behavior testable.
04

A project I enjoyed and what I learned from it

I enjoy building aesthetic and scalable mobile applications where the product feels smooth, polished, and reliable. One thing I pay close attention to is FPS, or frames per second. Apple generally targets 60 FPS for fluid interaction, so when I build animations, transitions, drawing surfaces, or other touch-heavy flows, I think carefully about frame pacing, JS-thread work, rendering cost, and memory usage. That attention to performance helps the app feel fluid and responsive for the user instead of just functionally complete.
