02

An example of a touch-heavy or interactive feature I built

Smooth Scribble is a focused version of that exact problem: fast finger movement, noisy touch points, and a drawing surface that needs to feel immediate instead of laggy.

  • The canvas runs inside a WebView-backed HTML5 Canvas so high-frequency drawing work stays off the React Native render loop.
  • The host app sends only low-frequency commands such as brush changes, undo, clear, and status updates.
  • The UI includes full-screen drawing, brush colors, brush widths, stroke count, undo, clear, and branded launch polish.
  • The smoothing logic is custom and lives in the project source, not inside Skia or a drawing library.
03

How I would improve a laggy drawing experience on older iPhones

  • First I would profile real devices and separate input latency, rendering cost, bridge traffic, memory pressure, and JS thread work.
  • I would keep pointer processing close to the drawing surface instead of sending every touch point through React Native state.
  • I would thin duplicate points, batch drawing with requestAnimationFrame, use coalesced pointer events when available, and cap stroke history.
  • I would tune smoothing constants against real touch latency so fast strokes do not trail behind the finger.
  • If stroke volume, export quality, or long sessions became the bottleneck, I would evaluate a native renderer while keeping the smoothing behavior testable.
04

A project I enjoyed and what I learned from it

I enjoy projects where product feel matters as much as functional correctness. Smooth Scribble reinforces the same lesson: the implementation is only good if the touch response feels natural in hand. Small details like event batching, brush feedback, safe-area-aware full-screen mode, and launch polish make the difference between a prototype and something that feels ready to evaluate.

Technical fit

How this maps to the requested app work

The exercise asks for a simple drawing canvas, but the choices below are the same choices I would care about in a production React Native app with canvas-heavy interaction.

01

React Native and TypeScript foundation

The project is a React Native CLI iOS app written in TypeScript, with a focused screen structure and small typed interfaces for canvas commands, brush state, and status messages.

02

Custom smoothing instead of a graphics library

The smoothing uses point thinning, velocity-aware low-pass filtering, and quadratic midpoint curves. That keeps the implementation inspectable and satisfies the exercise requirement to avoid drawing libraries such as Skia.

03

Responsive drawing loop

Fast pointer moves are batched through requestAnimationFrame, coalesced input events are consumed when available, and completed strokes are replayed only when needed for undo, clear, resize, or history trimming.

04

Bridge-conscious architecture

The React Native layer does not receive every touch point. Drawing stays inside the WebView canvas, while native UI controls send compact commands for brush changes, undo, clear, and status updates.

05

Mobile polish and release awareness

The app includes safe-area handling, full-screen canvas mode, icon controls, branded splash assets, an app icon, linting, type checking, and a small Jest smoke test.

Production thinking

What I would improve next

  1. Profile on older iPhones Measure input latency, frame pacing, memory growth, and WebView rendering behavior on real hardware before changing architecture.
  2. Tune smoothing constants against real touch Adjust point distance, force-accept timing, and velocity alpha until slow lines feel stable and fast scribbles stay close to the finger.
  3. Harden long-session behavior Improve memory policy for many strokes, large canvases, undo depth, resize replay, and possible future export flows.
  4. Add production drawing features only when scoped Layer persistence, export/share, pressure handling, eraser tools, and collaboration should be added only if they support the actual product need.
  5. Escalate renderer only if the data proves it If stroke volume or export quality outgrows the WebView canvas approach, I would move the renderer closer to native while preserving the custom smoothing model.
